Predictions of
ICU Mortality

Will the patient survive the hospitalisation stay?

The development of methods for prediction of mortality rates in the Intensive Care Unit (ICU) has been motivated by the need to to increase treatment effectiveness, improved care in ICU, and functional discharge to post-acute care facilities.

This project is obtained from the PhysioNet/CinC Challenge 2012. The main aim is to develop a solution for patient-specific prediction of in-hospital mortality that serves as a decision support for medical practitioners and medical care givers.

Feature Engineering

Each patient record is in the form of a time series - variables may be observed once, more than once, or not at all in some cases within 48 hours. Descriptor variables are recorded at time 00:00. To capture the best property of the time series, we will have to extract out batches of aggregated information from it. Table 1 below shows the original time series data.


Table 1

We have chosen to extract the mean, minimum and maximum, first and last observed data to represent the average and extremities of the time series. Table 2 below shows the extracted data for a patient record.


Table 2

After extraction, we observed that most patient records lack laboratory test results and most variables leading to many ‘Not-a-number’ (NaN) values in the dataset which would cause a problem in our modelling.

To resolve the issue of NaN values, we imputed the NaN values for each variable of each patient based on the median of the same age group, ICU type and gender group. This seems to better represent the patient’s condition. Certain conditions may have some extremes in certain variables, it would not be fair if we were to perform a simple mean or median imputation.

The box plot above visualises the outliers in the dataset. There are clearly some distinct outliers that doesn't look right. Let's investigate further into it.



Looking in depth in to the .describe attribute (from pandas), we made the following observations:

Urine output
Urine output seems to be the one that has the highest outlier. A higher urine output level could be due to a longer interval between data record and collection, hence the large discrepancy. (Urine output with a 10 hour interval will have much higher value than a 1 hour interval.) Urine output will not be a suitable feature for prediction. We will drop this feature.

PH
PH level is between 0 - 14. There are some records that are off the charts, probably due to human error or different metrics. We will be imputing them with the median of the same age group, ICU type and gender group.

Data Standardisation

After preprocessing, we used sklearn’s RobustScaler to standardise the numerical variables. Standardisation is essential to improve the predictive performance of our algorithms - especially so for gradient-based classifiers. RobustScaler scales the data based on the percentiles rather than the mean and standard deviation, it is not influenced by marginal outliers.

For categorial features, we used pandas’ pd.get_dummies method to obtain a matrix of dummy variables in 0 or 1. While some algorithms (such as decision trees and random forest) do not require this step, most machine learning algorithm requires all features to be numerical.

It is written in a class to allow easy implementation of sklearn’s convenient Pipelines and FeatureUnion.

Feature Selection



Recall from the correlation matrix that there are some features that are multicollinear with each other. The existence of multicolinear features may add noise to our model resulting in overfitting. Using the variance inflation factor(VIF), we will be able to estimate how much the variance of a regression coefficient of a particular feature is inflated due to multicollinearity. 

The VIF ranges from 1 (1 being not correlated). We have set a threshold of 5, any features with VIF > 5 will be eliminated from the model.

×

Imbalanced Class Resampling

Recall that we are dealing with an imbalanced dataset in this project. There are only 13.9% of in-hospital mortality (positives) in the dataset.



If we were to build a classifier in this dataset, we will probably obtain a 86% accuracy. However, the ‘correctly classified data’ would all come from the majority class. This is because most algorithms do not perform well with imbalanced datasets. It will tend to be bias towards the majority class.

To deal with this problem, there are two approaches that we have used in this dataset: Oversampling with Synthetic Minority Over-sampling Technique (SMOTE) and Edited Nearest Neighbours (ENN).

The SMOTE technique creates synthetic data on the minority class by finding the K-nearest neighbour (KNN), choses a random ‘neighbour’ and create a similar datapoint as the ‘neighbour’ randomly.

ENN removes datapoint if the datapoint’s class is different from the majority of it K nearest neighbours.

After resampling, we will increase the percentage of minority class from 13.9% to 38.
